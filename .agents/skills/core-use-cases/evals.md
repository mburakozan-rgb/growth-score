# Self-Grading Checklist (evals.md)

Use the following checks to grade your output before presenting it. For each check, rate it as PASS or FAIL. If any check fails, regenerate the section.

## Pass/Fail Evaluation Criteria

1. **Independent Check**:
   - *Requirement*: Each use case in the table is deployable without shipping another use case first.
   - *Verdict*: [PASS/FAIL]

2. **End-to-End Check**:
   - *Requirement*: Every use case maps a complete journey from a clear trigger to a valuable user outcome. No "partial work" epics (e.g., "Build SQLite schema" is NOT a usecase).
   - *Verdict*: [PASS/FAIL]

3. **Audience-Specific Check**:
   - *Requirement*: Every use case specifies the exact audience (e.g., "Strategy Consultant" or "Curator") rather than a generic "User".
   - *Verdict*: [PASS/FAIL]

4. **Testable Check**:
   - *Requirement*: Clear, numerical or binary acceptance criteria exist for every use case (e.g., "Must generate a .md file inside /standards").
   - *Verdict*: [PASS/FAIL]

5. **3-Week Cycle Check**:
   - *Requirement*: The complexity of each use case is small enough to fit a single 3-week cycle. Large epics must be explicitly split.
   - *Verdict*: [PASS/FAIL]

6. **Suggested Epic Mapping Check**:
   - *Requirement*: A sequencing timeline is proposed, starting with foundational capability builders and ending with presentation/export layers.
   - *Verdict*: [PASS/FAIL]
